# 로케이션 슬롯 바코드 라벨 + 입고 시 스캔 배정 — 설계

## 배경

파렛트랙의 각 슬롯(단×좌우)에 바코드가 포함된 라벨을 출력해서 랙에 붙일 수 있게 하고, 입고지시 단계에서 그 바코드를 스캔해서 특정 파렛트를 원하는 슬롯에 직접 배정할 수 있게 한다. 현재는 파렛트 자체에 대한 바코드 라벨(`LabelPrinter.js`)만 있고, 랙/슬롯에 붙이는 라벨은 전혀 없다.

## 현재 구조 (참고)

- 슬롯 = `location_id` + `tier`(1~4) + `side`('L'/'R') 조합. DB에 슬롯 자체의 행은 없고, `pallets` 테이블에 어느 슬롯이 점유됐는지만 기록됨
- `locations.code`는 랙(=location 1행) 단위로 유일한 문자열(예: `A-01`). 슬롯 단위 식별자는 현재 어디에도 저장되어 있지 않음
- 바코드는 이미 `jsbarcode`로 3곳(`LabelPrinter.js`, `inbound/page.js`, `work-orders/page.js`)에서 CODE128 형식으로 파렛트 코드/작업지시 번호를 인쇄 중 — 같은 패턴(`useRef` + `useEffect` + `JsBarcode(svgEl, value, {...})`)을 그대로 재사용
- `src/components/BarcodeInput.js`: 스캐너(키보드 방식) 입력을 받는 컴포넌트가 이미 구현되어 있으나 **어디서도 쓰이지 않고 있음** — Enter 시 `onScan(code)` 콜백 호출
- 입고 흐름은 3단계 탭(등록→지시→완료) 위저드. **지시(Instruct) 단계**에서 구역(zone)을 고르면 그 구역의 빈 슬롯을 앞에서부터 순서대로 자동 배정하는 로직만 있고(`handleZoneChange`, `src/app/inbound/page.js`), 특정 슬롯을 수동으로 지정하는 기능은 없음
- 자동 배정 결과(`assignment` 배열: `{locationId, locationCode, tier, side}` 파렛트당 1개)는 로컬 상태에 미리보기로만 있다가, "지시 완료" 클릭 시 한 번에 `pallets` 테이블에 insert됨
- `availableSides(slot_config)`(어느 side가 사용 가능한지) 로직이 `RackModal.js`, `inbound/page.js`, `zone/[zoneCode]/page.js` 3곳에 이미 중복되어 있음 (이번 기능에서 4번째 사용처가 생기지만, 기존 3곳을 공용 모듈로 묶는 리팩터링은 이번 범위 밖 — 별도로 설계된 "개별 슬롯 사용 설정" 작업에서 다루기로 함)

## 요구사항

1. 슬롯 하나당 바코드 라벨을 출력할 수 있다. 라벨에는 슬롯을 식별하는 코드(랙 코드+단+좌우)와 그 값을 인코딩한 바코드가 포함된다
2. 랙 하나를 선택하면 그 랙의 사용 가능한 슬롯(최대 8개, `slot_config` 반영) 전체 라벨을 한 번에 출력할 수 있다
3. 입고지시 단계에서, 자동 배정된 파렛트별 슬롯을 스캔으로 다른 특정 슬롯으로 덮어쓸 수 있다
4. 스캔한 슬롯이 이미 다른 파렛트(재고 또는 이번 배치의 다른 줄)에 배정되어 있으면 덮어쓰지 않고 오류를 안내한다
5. 스캔한 코드가 존재하지 않는 랙/슬롯이거나 형식이 올바르지 않으면 오류를 안내한다

## 슬롯 코드 형식

`{location.code}-{tier}{side}` — 예: `A-01-4L` (A-01 랙의 4단 좌측). 파싱은 정규식 `/^(.+)-(\d)([LR])$/`로 마지막 `-{숫자}{L|R}` 부분을 분리한다 (location.code 자체에 하이픈이 들어있어도 안전 — 마지막 하이픈 뒤만 슬롯 접미사로 취급).

## 1) 라벨 출력

- `src/app/locations/page.js`의 랙 목록 각 행에 "슬롯 라벨 출력" 버튼 추가
- 클릭 시 그 랙의 `availableSides(slot_config)` × `TIERS`로 계산된 사용 가능한 슬롯 전체(최대 8개)를 대상으로, 기존 `LabelPrinter.js`/입고완료 단계의 `BatchLabelModal`과 동일한 패턴(여러 장을 한 번에 `window.print()`로 출력, `#wms-label-print`/`printing-label` 관례 재사용)으로 새 모달을 띄운다
- 라벨 1장 구성: 슬롯 코드(`A-01-4L`, 크게) + CODE128 바코드(슬롯 코드 인코딩) + 랙 코드/단/좌우 텍스트("A-01 · 4단 · 좌측")
- `slot_config`로 막힌 side는 라벨 대상에서 제외 (예: `slot_config='L'`이면 우측 4개는 출력하지 않음)

## 2) 입고 시 스캔으로 위치 지정

- `src/app/inbound/page.js`의 `InstructModal` — 자동 배정 결과 미리보기 테이블(현재는 읽기 전용 텍스트만 표시)의 각 줄에 `BarcodeInput`을 추가
- 스캔 시:
  1. 코드를 정규식으로 파싱해 `locationCode`, `tier`, `side` 추출 — 형식이 안 맞으면 오류("올바른 슬롯 코드가 아닙니다")
  2. `locationCode`로 `locations` 테이블 조회 — 없으면 오류("존재하지 않는 로케이션입니다")
  3. 해당 슬롯이 그 랙의 `slot_config`로 허용된 side인지 확인 — 아니면 오류("이 랙에서 사용할 수 없는 슬롯입니다")
  4. 이미 점유된 슬롯인지 확인 — 기존 점유 조회(`occSet`, 이미 계산되어 있는 `stored`/`pending` 파렛트 집합)와 **이번 배치 내 다른 줄에 이미 배정된 슬롯**(=`assignment` 배열의 다른 항목) 둘 다 확인 — 겹치면 오류("이미 사용 중인 슬롯입니다")
  5. 통과하면 해당 줄의 `assignment[i]`를 스캔한 `{locationId, locationCode, tier, side}`로 교체
- 자동 배정 로직(`handleZoneChange`)은 그대로 유지 — 스캔은 배정된 결과에 대한 줄 단위 덮어쓰기 기능으로 추가

## 엣지 케이스

| 상황 | 처리 |
| --- | --- |
| 같은 슬롯을 두 번 스캔(같은 배치 내 중복) | 두 번째 스캔은 "이미 사용 중인 슬롯입니다" 오류로 거부 |
| 스캔으로 덮어쓴 후 구역(zone)을 다시 선택 | `handleZoneChange`가 `assignment`를 초기화하므로 스캔 결과도 함께 사라짐 — 기존 동작과 동일한 특성, 별도 보존 로직 없음 |
| 랙에 슬롯이 하나도 없는 상태(`slot_config`로 전부 막힘)에서 라벨 출력 클릭 | "출력 가능한 슬롯이 없습니다" 안내, 모달 띄우지 않음 |
| 스캔한 슬롯이 다른 구역(zone)의 랙에 속함 | 허용함 — 입고지시는 구역 선택 없이도 스캔만으로 슬롯을 특정할 수 있으므로, 스캔된 슬롯의 실제 소속 구역과 무관하게 배정 가능 (요구사항에 구역 제한이 없으므로 막지 않음) |

## 범위 밖 (Out of scope)

- 슬롯을 열 때 카메라로 스캔(`BarcodeInput`이 지원하는 `BarcodeDetector` 기능)을 강제하거나 안내하는 UX는 다루지 않음 — 키보드형 스캐너(Enter로 제출)를 기본 가정
- `availableSides` 로직을 공용 모듈로 묶는 리팩터링은 다루지 않음 (별도 "개별 슬롯 사용 설정" 작업에서 다룰 예정)
- 출고(outbound)/피킹 단계에서의 스캔 활용은 다루지 않음 — 이번은 입고지시 단계로 한정
- 슬롯 라벨의 mm 단위 정밀 사이즈(다이컷 라벨 용지) 대응은 다루지 않음 — 기존 파렛트 라벨과 동일하게 일반 프린터용 페이지 레이아웃 사용
